## 1. Project Overview

**A1Care** is a **home healthcare platform** that connects patients with doctors and home-care service providers (nurses, physiotherapists, etc.). The platform has three distinct user sides:

| App | Status |
|-----|--------|
| User App (Patient) | ⚠️ **Directory exists but is EMPTY — not yet built** |
| Partner App (Provider/Staff) | ⚠️ **Directory exists but is EMPTY — not yet built** |
| Admin Panel | ⚠️ **Directory exists but is EMPTY — not yet built** |
| **Backend API (`a1care_serverV2`)** | ✅ **Fully built and running** |

The **only live code in this project is the backend**. All three frontend applications (user app, partner app, admin panel) are yet to be developed.

## 5. Module-by-Module Breakdown

### 5.1 Patient Authentication — `/api/patient/auth`

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|
| POST | `/send-otp` | ❌ | Generates 6-digit OTP → sends via Twilio → stores in Redis (`otp:{mobileNumber}`, TTL 300s) |
| POST | `/verify-otp` | ❌ | Reads OTP from Redis, validates, creates Patient record if new, returns JWT |
| GET | `/profile` | ✅ | Returns patient document + populated `primaryAddressId` |
| PUT | `/profile` | ✅ | Validates with Zod, uploads profile image to S3, updates patient document (`isRegistered=true`) |

**Patient Model Fields:**

| Field | Type | Notes |
|-------|------|-------|
| `mobileNumber` | Number | Unique, required |
| `name` | String | — |
| `email` | String | Lowercase |
| `profileImage` | String | S3 URL |
| `gender` | Enum | Male / Gender / Other |
| `dateOfBirth` | Date | — |
| `fcmToken` | String | Push notifications (future) |
| `isRegistered` | Boolean | false until profile filled |
| `primaryAddressId` | ObjectId | → `patient_addresses` |

> ⚠️ **Security Issue:** OTP `123123` is hardcoded as a bypass in `verifyOtpForPatient`. Must be removed before production.

---

### 5.2 Doctor / Staff — `/api/doctor`

Two separate auth flows based on security level:

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|
| POST | `/auth/send-otp` | ❌ | Generates OTP → HMAC hashes it → stores in Redis as `otp:{sessionId}` with attempts counter |
| POST | `/auth/verify-otp` | ❌ | Verifies HMAC match, max 5 attempts, returns JWT (7-day expiry) |
| GET | `/auth/details` | ✅ | Returns authenticated staff document |
| POST | `/auth/otp/status` | ❌ | Checks if OTP session still live in Redis |
| PUT | `/auth/register` | ✅ | Completes staff profile (`isRegistered=true`) |
| POST | `/create` | ❌ | Admin: creates doctor record directly |
| GET | `/:doctorId` | ❌ | Get doctor by ID |
| GET | `/staff/role/` | ❌ | Get staff by comma-separated role IDs |

**Key security difference vs Patient auth:** Doctor OTP is HMAC-SHA256 hashed + stored per UUID session (not phone number key), with 5-attempt lockout.

**Doctor Model Fields:** `mobileNumber`, `name`, `gender`, `startExperience`, `specialization[]`, `about`, `workingHours`, `consultationFee`, `status` (Pending/Active/Inactive), `rating`, `roleId`, `isRegistered`

---

### 5.3 Doctor Availability & Slots — `/api/doctor`

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|
| POST | `/slot/create/` | ✅ | Sets doctor recurring availability (`startingTime`, `endingTime`, `slotDuration`) |
| POST | `/slot/block/:doctorId` | ❌ | Blocks specific date/time or weekday |
| GET | `/slots/:date` | ✅ | Computes available slots for a date |

**Slot computation logic (`availableSlotByDoctorId`):**
1. Fetch doctor's `DoctorAvailability` record (start, end, duration)
2. `generateSlots()` — creates all possible time slot intervals
3. Fetch `BlockTime` records matching date OR weekday
4. Fetch `DoctorAppointment` records for that date (already booked)
5. Filter: remove any slot overlapping busy intervals
6. If today: also remove past slots
7. Return readable `{ startingTime, endingTime }` array

### 5.4 Roles — `/api/role`

Roles represent **staff specialisation types** (e.g. Nurse, Physiotherapist, Cardiologist).

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|
| POST | `/` | ❌ | Create a new role |
| GET | `/` | ❌ | List all roles |

**Role Model:** `name`, `description`

Roles are the **core matching key** — `ChildService.allowedRoleIds[]` determines which staff roles can see and accept a given service request.

### 5.5 Service Catalogue — 3-Level Tree

```
Service  (top-level category)
  └── SubService  (linked by serviceId)
        └── ChildService  (linked by serviceId + subServiceId)
              └── allowedRoleIds[]  ← which staff can fulfill this
                  price, imageUrl, description
```

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|

**SubServices — `/api/subservice`**

| Method | Endpoint | Auth | What it does |

### 5.6 Doctor Appointments — `/api/appointment`

Patient books a specific doctor at a specific time slot.

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|
| POST | `/booking/:doctorId` | ✅ | Creates `DoctorAppointment` (status: Pending) |
| GET | `/patient/appointments` | ✅ | Patient's all appointments (populated doctor + patient) |
| GET | `/patient/appointments/pending` | ✅ | Doctor sees their Pending + Confirmed appointments |

**Appointment fields:** `patientId`, `doctorId`, `date`, `startingTime`, `endingTime`, `status` (Pending/Confirmed/Completed/Cancelled)

---

### 5.7 Service Bookings (Home Care) — `/api/service/booking`

Core marketplace flow — patient requests home care, providers accept/reject.

| Method | Endpoint | Auth | What it does |
|--------|----------|------|-------------|
| POST | `/create/` | ✅ | Patient creates service request (max 2 concurrent) |
| GET | `/user` | ✅ | Patient gets all their service requests |
| GET | `/get/pending` | ✅ | Patient gets active requests (PENDING/BROADCASTED/ACCEPTED/IN_PROGRESS) |
| GET | `/request/:requestId` | ✅ | Get specific request details |
| GET | `/provider/:roleId` | ✅ | Provider gets requests matching their role |
| POST | `/accept/:serviceRequestId` | ✅ | Provider accepts → status → ACCEPTED, creates ServiceAcceptance |
| POST | `/reject/:serviceRequestId` | ✅ | Provider rejects → creates ServiceAcceptance (REJECTED) |

**Service Request Status Machine:**

```
[Patient Creates] → PENDING
    → BROADCASTED (broadcast to providers)
    → ACCEPTED    (provider accepts)
    → IN_PROGRESS (service started)
    → COMPLETED   (service finished)
    → CANCELLED   (patient cancels / no takers)
```

**Business Rule:** Max **2 concurrent active** service requests per patient.

**Matching Logic:** `ChildService.allowedRoleIds` contains role IDs → when provider queries with `roleId`, backend finds all child services with that role → returns matching PENDING requests.

## 9. Data Flow Summary

### User Booking a Home Service
```
Patient sends OTP → verifies → gets JWT
→ Adds address
→ Browses Services → SubServices → ChildServices
→ Creates ServiceRequest (PENDING, max 2 active)
→ Provider (matching roleId) sees request
→ Provider accepts → ServiceRequest.status = ACCEPTED
→ Service happens → IN_PROGRESS → COMPLETED
```

### Patient Booking a Doctor
```
Patient sends OTP → verifies → gets JWT
→ Views doctor profile
→ Gets available slots for a date
→ Books slot → DoctorAppointment (PENDING)
→ Doctor confirms → Confirmed
→ Appointment completes → Completed
```

## 11. Summary

The **A1Care backend is a well-structured, modular Node.js/TypeScript/Express API** for a home healthcare marketplace. The core booking and authentication flows are implemented and functional. However:

- All three frontend apps (user, partner, admin) **do not exist yet**
- Real-time features (Socket.IO) are installed but **not connected**
- Several status-update flows (complete, cancel, in-progress) are **not yet implemented**
- A **security bypass (OTP 123123)** must be removed before production
- Several routes that should be admin-protected **have no auth guard**

The backend is at roughly **60–65% completion** for an MVP.
